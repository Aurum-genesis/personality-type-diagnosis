# Socionics Publish

Ты — помощник по проекту «Персональная диагностика личности на основе соционической типологии». Репозиторий: `zerocoder/socionica/practice/online-service/publish` (опубликован как `Aurum-genesis/personality-type-diagnosis`).

## Где что лежит

- `index.html` (≈104 КБ, 1538 строк) — всё в одном файле: HTML + CSS + JS
- `pdf_data.js` (≈91 КБ, 1 строка) — минифицированный base64 PDF, который отдаётся кнопкой «📥 PDF-инструкция»
- `Календарь_упражнений.xlsx` (≈28 КБ) — Excel-трекер упражнений, лежит рядом с `index.html` и скачивается кнопкой «📥 Excel-трекер»
- `images/s1.PNG` — скриншот результата, используется в `README.md`
- `README.md` — описание проекта, что внутри, как запустить
- `.gitignore` — стандартный (мусор редакторов и ОС)

**Внимание:** в проекте есть папка `images/` с двумя файлами — `s1.PNG` (≈357 КБ) и `im` (2 байта, похоже на артефакт). Не удалять `im` без явного указания пользователя.

## Архитектура index.html — карта файла

Файл `index.html` устроен так (по строкам):

1. **CSS (строки 7–359)** — тёмная тема, CSS-переменные `:root`, анимации `@keyframes`. Содержит стили для:
   - стартового экрана (`#start`, `.start-block`, `.btn-primary`)
   - прогресс-кольца (`.progress-ring`, `.ring-fill`)
   - орбиты функций на экране результата (`.orbit-wrap`, `.orbit-node`, `.orbit-link`)
   - модального окна плана развития (`.modal-overlay`, `.fn-block`, `.level-btn`)
   - радар-диаграммы (canvas + CSS-анимация `.radar-anim`)

2. **HTML-структура (строки 361–410)** — 5 экранов, переключаются через `display: none/block`:
   - `#start` — стартовый экран с тремя раскрывающимися блоками и кнопкой «Начать диагностику»
   - `#typeSelector` — сетка 4×4 для выбора типа вручную
   - `#quiz` — контейнер для вопросов (заполняется JS)
   - `#progress` — кольцо прогресса (заполняется JS)
   - `#result` — контейнер для результата (заполняется JS)
   - Подключается `<script src="pdf_data.js">` ДО основного `<script>`

3. **Данные и константы (строки 412–529)** — JS-объекты:
   - `QUAD` — 4 квадры (alpha/beta/gamma/delta) с цветами
   - `TYPE_QUAD` — привязка кода типа к квадре (все 16)
   - `allTypes` — массив из 16 типов: `{ code, nick, full, ei, stack: [8 функций] }`
   - `typeDescriptions` — текстовое описание каждого типа
   - `fnTraits` — короткое описание каждой из 8 функций в сильной/слабой позиции
   - `levelData` — название, описание и 4 уровня развития для каждой функции
   - `levelNames` — `['Примитивный','Нормальный','Развитый','Творческий']`

4. **Вопросы теста (строки 533–754)**:
   - `ieQuestions` — 3 вопроса I/E
   - `snQuestions` — 3 вопроса S/N
   - `tfQuestions` — 3 вопроса T/F
   - `block2Questions` — словарь из 8 групп (по парам кандидатов), в каждой ровно 7 вопросов:
     - 2 парных (`type:'pair'`)
     - 1 ассоциативный (`type:'association'`, 4 варианта)
     - 1 на болевую (`type:'bolevaya'`, 2 варианта)
     - 1 на суггестивную (`type:'sugestivnaya'`, 2 варианта)
   - Ключи групп: `'ИЛИ-ЛИИ'`, `'ИЭИ-ЭИИ'`, `'ИЛЭ-ЛИЭ'`, `'ИЭЭ-ЭИЭ'`, `'ЛСИ-СЛИ'`, `'СЭИ-ЭСИ'`, `'ЛСЭ-СЛЭ'`, `'СЭЭ-ЭСЭ'`
   - Функции-помощники: `isSensorType`, `isIntuitiveType`, `isLogicType`, `isEthicType`, `getGroupKey`

5. **Логика теста (строки 756–1129)**:
   - Состояние: `step` (1..4), `qIndex`, `scoreI/E/S/N/T/F`, `candidates`, `stepResults`, `block2QuestionsList`, `block2Scores`, `block2Key`
   - `startTest()` — сброс, показать `#quiz`
   - `showTypeSelector()` — показать сетку 4×4 из `allTypes`
   - `showQuickResult(code)` — мгновенный показ результата по коду
   - `renderQuestion()` — рисует текущий вопрос для шага 1/2/3, иначе зовёт `renderBlock2Question()`
   - `renderProgress(...)` — кольцо прогресса + шаги `I/E → S/N → T/F → Block 2`
   - `answer(choice)` — обработка выбора; на блоке 2 передаёт `choice` = код победителя
   - `endStep()` — фильтрует `candidates` после каждого шага дихотомии и показывает промежуточный результат
   - `showIEStepResult`, `showSNStepResult`, `showTFStepResult` — промежуточные экраны
   - `goToStep2/3`, `goToBlock2` — переходы
   - `renderBlock2Question`, `answerBlock2`, `showBlock2Result` — сам блок 2

6. **Результат (строки 1131–1224)**:
   - `buildResultHTML(type)` — собирает HTML: код типа, ник, описание, орбита, таблица функций, кнопки
   - `buildOrbitHTML(type, posLabels)` — 8 узлов вокруг центрального круга, SVG-связи
   - Позиции функций: 0..7 = «базовая, творческая, социальная маска, болевая, внушение, ободряющая, игнорируемая, демонстративная»
   - Сильные позиции: `i < 2 || i > 5` (то есть 1, 2, 7, 8) — зелёные
   - Слабые: 3, 4, 5, 6 — красные

7. **План развития (строки 1226–1477)**:
   - Состояние: `planLevels = {}`, `planType`
   - `showPlan(code)` — открывает модалку, рендерит 8 блоков (по 4 кнопки уровня в каждом)
   - `selectLevel(fn, level)` — выставляет уровень для функции
   - `showChart(code)` — рисует радар на canvas 360×360, центр (180,180), R=140, 8 осей
   - `generatePlan()` — генерирует план на 4 месяца по сильным функциям (позиции 1, 2, 7, 8)
   - Радар: значение = `planLevels[fn] + 1` (от 1 до 4), `+1` потому что 0 = «не выставлено»

8. **Скачивание (строки 1479–1501)**:
   - `downloadChartImage()` — сохраняет canvas как `график_развития.png`
   - `downloadFile(filename)`:
     - `Упражнения_по_функциям.pdf` → подставляет `pdfDataUrl` из `pdf_data.js`
     - всё остальное (включая `Календарь_упражнений.xlsx`) → ставит `a.href = filename` (относительный путь, файл должен лежать рядом с `index.html`)

9. **HTML модалки (строки 1512–1535)**:
   - `#planModal` — модальное окно плана
   - Внутри: `#planBody`, `#chartSection`, `#radarChart` (canvas), `#chartLegend`, `#planResult`
   - Кнопки: «Составить план», «Сохранить изображение», «📥 Excel-трекер», «📥 PDF-инструкция»

## Как запустить локально

Проект — статический одностраничник. Никаких сборок, зависимостей, сервера.

**Самый простой способ:** двойной клик по `index.html` — откроется в браузере через `file://`.

**Если нужна живая перезагрузка при правке:** VS Code + расширение Live Server, корень = папка `publish/`.

**Если открывать через `http://`:** любой статический сервер в корне папки. Но помнить: на GitHub Pages Excel и PDF могут отдаваться криво, потому что хостинг не всегда выставляет правильный MIME для `.xlsx`.

## Как читать и редактировать

- **Цвета и тема** — в `:root` (строки 8–21). Поменять `--accent` или `--green`/`--red` меняет цвета по всему проекту.
- **Тексты типов** — объект `typeDescriptions` (строки 448–465). Прямо менять строки.
- **Вопросы теста** — `ieQuestions`, `snQuestions`, `tfQuestions` (строки 548–566) и `block2Questions` (строки 569–754). У каждого вопроса — поля `q`, `a/b` или `options`, у вариантов — `text`, `fn` (код функции), `win` (код победителя).
- **Добавить новую пару кандидатов в блок 2** — добавить ключ в `block2Questions` вида `'КОД1-КОД2'` (отсортированы через `getGroupKey`).
- **Список типов** — массив `allTypes` (строки 429–446). У каждого типа `stack` всегда ровно 8 функций.
- **Уровни функций** — `levelData` (строки 480–529). Внутри каждой функции массив `levels` из 4 строк-описаний.

## Частые задачи

### Поменять вопросы теста

1. Прочитай `index.html` (нужный диапазон строк) через `Read`.
2. Покажи пользователю текущий текст вопроса.
3. Спроси новый текст (или возьми из источника, если есть).
4. Правь через `Edit`, не `Write` (правило CLAUDE.md: не перезаписывать файлы).
5. Если меняешь вопросы блока 2 — следи, что `aKey/bKey` и `win` остаются консистентны (тип-`pair` имеет `a` и `b`, у каждого `fn` и `win`).

### Обновить README

1. Прочитай `README.md` через `Read`.
2. Покажи пользователю текущее содержимое.
3. Спроси, что именно поменять (не додумывать структуру).
4. Правь через `Edit`.
5. Проверь, что ссылки на скриншоты (`images/s1.PNG` и т.п.) совпадают с реальными файлами.

### Запустить проект

- Скажи пользователю: двойной клик по `index.html`.
- Если хочет через VS Code Live Server — это нормальный вариант, корневая папка = `publish/`.
- Не предлагать `npm`, `pip`, сборщики — их в проекте нет и не должно быть (по дизайну).

### Проверить ошибки

Самые частые места:

1. **Битый вопрос в `block2Questions`** — если в группе меньше 7 вопросов или у вопроса `type:'pair'` нет `a` и `b`, при `renderBlock2Question` будет `undefined.q` или `q.a.text` упадёт.
2. **Несовпадение ключа** — `getGroupKey(a, b)` всегда сортирует. Если добавил новый ключ, проверь порядок: `getGroupKey('ЛИИ','ИЛИ')` = `'ИЛИ-ЛИИ'`, а не `'ЛИИ-ИЛИ'`.
3. **`fn` не из 8** — `fnTraits[fn].name` и `levelData[fn]` упадут, если в `type.stack` или в `options` указана функция не из `{Ti,Te,Ni,Ne,Si,Se,Fi,Fe}`.
4. **Кириллица в `.xlsx`/`pdf`** — если кто-то правит `Календарь_упражнений.xlsx` или `pdf_data.js` через редактор, который не поддерживает UTF-8, кириллица станет кракозябрами в браузере.
5. **Excel-кнопка не скачивает** — файл `Календарь_упражнений.xlsx` должен лежать в той же папке, что и `index.html`. Через `file://` это работает; через некоторые HTTP-серверы может не работать (нужен правильный MIME).

### Подготовить релиз

1. Проверить, что `index.html` валиден (нет `undefined`, все 16 типов на месте, все 8 групп в `block2Questions`).
2. Проверить, что `README.md` отражает текущее состояние (количество вопросов, реализованные блоки).
3. Проверить, что `images/s1.PNG` существует и не пустой.
4. Убедиться, что `.gitignore` не закоммичен (это служебный файл, но иногда полезно его видеть в репо).
5. Сделать `git status` — посмотреть, что изменилось, прежде чем коммитить.
6. `git add .` → `git commit -m "…"` → `git push origin main` (или через `gh repo sync`).
7. Если меняется описание репозитория — `gh repo edit --description "…"`.

### Публикация через GitHub CLI

Команды, которые могут пригодиться:

- `gh auth status` — проверить, залогинены ли
- `gh repo view Aurum-genesis/personality-type-diagnosis` — посмотреть состояние репо
- `gh repo create <name> --public --source=. --push` — создать и запушить
- `gh repo edit --description "…"` — поменять описание
- `gh api repos/<owner>/<repo>/contents --jq '.[].name'` — список файлов в репо

## Правила работы (по CLAUDE.md)

- **НЕ перезаписывать** существующие файлы через `Write`. Только `Edit`.
- **НЕ удалять** `images/im` без явного указания (это файл из репо, не моё содержимое).
- **НЕ додумывать** содержимое вопросов, описаний типов, упражнений в Excel. Спрашивать.
- **Перед правкой** прочитать файл целиком (или релевантный кусок) через `Read`.
- **Перед коммитом** проверить `git status` и `git diff` — что именно уезжает.
- **Excel-файлы** — никогда не править через openpyxl/аналоги. Только читать для проверки. Для правок — показать пользователю структуру в чате и спросить.
- **PDF** — `pdf_data.js` менять только если пользователь явно скажет «перегенерируй PDF».
- **Цитаты из источника** — `typeDescriptions`, `levelData`, `fnTraits` пришли из соционической методички. Если меняешь — спросить пользователя, не выдумывать формулировки.

## Что НЕ нужно делать

- Не предлагать переписать проект на React/Vue. Это осознанное решение — vanilla JS.
- Не предлагать бэкенд, базу данных, регистрацию. В описании проекта это явно отложено на следующие этапы.
- Не предлагать npm-установку SheetJS или других библиотек для Excel — текущая версия скачивает файл напрямую, и это работает через `file://`.
- Не додумывать «логичную» формулировку вопроса. Если пользователь говорит «поменяй вопрос 5 в блоке 2» — спроси, на что.
